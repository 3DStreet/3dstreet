# 005 — Shift+LB decoupled-view sub-plan: adversarial review

*Working draft 2026-05-11.*

Review of `claude/specs/001-shiftrotate-decoupled-view.md` (the bug-fix sub-plan that replaces `camera.lookAt(center)` with dual-spherical rotation). Headline: the plan is sound. Math round-trips cleanly, the "no snap at gesture start" claim holds, and the rotate-in-place subsumption is correct. Findings below, ordered by significance.

## Findings

- **The "behaviour identical when starting aimed at centre" claim is correct, but only because the spherical decomposition for two same-direction vectors is the same up to radius.** Trace: if `viewDir = normalize(center − pos)`, then `−viewDir = normalize(pos − center) = normalize(offsetPos)`, so `offsetView` and `offsetPos` share direction. Their `Spherical` representations have identical `(theta, phi)` and differ only in `radius`. Apply the same `(dTheta, dPhi)`: both rotate to a new shared direction `D'`. New `pos = center + radiusPos * D'`; `lookTarget = newPos − 1 * D' = center + (radiusPos − 1) * D'`, which is collinear with `(center − newPos)`. `camera.lookAt(lookTarget)` aims at centre. **Result: identical to the current `lookAt(center)` orbit.** Worth re-stating explicitly in the spec — the claim is currently asserted but not derived.

- **Zero-deltas round-trip is numerically clean.** `THREE.Spherical.setFromVector3` followed by `setFromSpherical` of a unit vector produces a residue of order `~1e-15`. With `dxPx = dyPx = 0`, `actualDPhi = 0` and `dTheta = 0`, so `sphView` is unchanged and `newOffsetView ≈ −fwd` to within float precision. `lookTarget = pos − newOffsetView ≈ pos + fwd`, which is exactly the point 1m along the current view direction — `lookAt` to that is a no-op. Same logic applies to `offsetPos`: `setFromVector3`/`setFromSpherical` round-trips `offsetPos` so `camera.position = center + newOffsetPos ≈ camera.position` (within ~1e-15 m). **No position OR orientation jump on the first move.** The "position-jump unlikely" hedge in the plan can be promoted to "position-jump is mathematically excluded".

- **The "off-centre orbit" feel risk in §Risks is the load-bearing UX trade-off, not a math concern.** With the new design, a user who Shift+LB-drags 180° of yaw from a position off to one side of the diorama ends up on the opposite side of the diorama *looking in the rotated direction* — i.e. probably looking off into the distance, not at the diorama. The current snap-and-orbit semantics would have them looking continuously at the diorama through the whole drag. The plan correctly identifies this and accepts it. **Feel-test should explicitly check this scenario.** Recommendation: add to §Risks "feel-test smoke item: stand 30m off the side of the diorama at street level, Shift+LB-drag a slow 180° yaw, verify the resulting camera pose feels intentional rather than disorienting." If it feels bad, the fallback is the hybrid in the next bullet.

- **A cheaper alternative not on the plan's list: "skip the lookAt on the first frame of the gesture".** Keep the existing `orbit + lookAt(center)` math; gate the `lookAt(center)` call behind "has the gesture moved by more than N pixels yet?". On the very first non-zero-delta move, snap-to-centre once (one small step, not a step-zero jolt), then continue orbiting. This fixes the *user-reported* snap (which is at gesture-start with zero deltas? or with the first tiny non-zero delta? — read the user report again: "the very first time I Shift+LB *and start to drag the mouse*") **without** changing the "always looks at centre during orbit" feel that the current design provides. The plan's design is a strict generalisation, which is mathematically clean but commits to a different orbit feel. Worth weighing this alternative — the user-reported bug might be a Phase-2-was-supposed-to-do-this-but-the-snap-is-too-aggressive issue, not a "we want decoupled view" issue. Recommend asking the user before committing to the dual-spherical design.

- **`offsetPos.phi` near the pole is a thin edge case but worth one sentence.** The plan's "no separate phi clamp" on position is fine *in normal poses* because `offsetPos.phi` is bounded by physical geometry (camera at eye-height orbiting centre at eye-height gives `phi ≈ π/2`). But if a user gets the camera nearly directly above or below the rotation centre (e.g. plan-view-ish pose with rule 2 centre at eye height — camera at y=200m, centre at y=1.5m, both at same xz → offsetPos along +Y → `phi ≈ 0`), then a `dTheta` rotates around an ill-defined azimuth and a `dPhi` derived from view-tilt can push offsetPos through the pole. `setFromSpherical` near `phi = 0` is well-defined (just scales `sin(phi) ≈ phi`) so there's no NaN, but the position can swing very sensitively. The plan should add a sentence: "position spherical near the +Y/−Y pole is left to spherical's natural degeneracy; in practice the user can't reach this without a near-vertical pose where the rotation centre is already Rule 1 or Rule 3."

- **The "rotate-in-place is now an automatic consequence" claim is mostly true, with one wrinkle.** When `offset.lengthSq() < 1e-6`, the plan's code path: `hasPositionOrbit = false`, so it skips the position math and only rotates the view direction. **Behaviour matches existing `rotateInPlace` branch exactly.** But: the current code also does `this.center.copy(camera.position)` in the rotate-in-place case (line 1103), whereas the plan's new code does `this.center.copy(center)` unconditionally. If `offsetPos.lengthSq < 1e-6` then `center ≈ camera.position` to within 1mm, so `this.center.copy(center)` and `this.center.copy(camera.position)` differ by sub-mm. Not visible. **Note this in the spec** for completeness — it's a benign drift but it changes `this.center` by sub-mm in the rotate-in-place case, and any future reader will wonder.

- **Test case 2 ("zero deltas, camera NOT aimed at centre") is the load-bearing regression check, and it's not trivially passing.** The current code's failure mode in this scenario is: `camera.position.copy(center).add(newOffset)` — newOffset comes from setFromSpherical of unchanged sphPos, so position is preserved — but `camera.lookAt(center)` then re-orients. So a test that asserts "orientation unchanged with zero deltas" *does* catch the current bug (current code reorients to centre; new code does not). The test is meaningful, not just a `setFromSpherical(setFromVector3(v)) === v` round-trip check. **Good.** Recommend the test asserts on the camera's *quaternion* or *forward vector* (the user-perceived orientation), not on intermediate Spherical fields, to avoid hiding the regression behind a math-level assertion that the buggy code would also pass.

- **`shiftRotateStep` helper signature is sufficient, but reading `camera` directly couples the helper to a `THREE.Camera`-shaped object.** Cleaner: pass `(pos, viewDir, center, dxPx, dyPx, speed) → {pos, lookTarget}`. Then the helper has no THREE coupling beyond `Vector3`/`Spherical`, matching the rest of `navMath.js`. The current proposed signature `(camera, center, ...)` works, but `camera.getWorldDirection` is the only thing it uses on the camera — easier to test with explicit `viewDir`.

- **Sign convention on `lookTarget` is correct.** `offsetView = −fwd` (unit, behind camera). After rotation, `newOffsetView` is the rotated "behind" vector. `lookTarget = pos − newOffsetView = pos + newForward`, a point 1m along the new view direction. `camera.lookAt(lookTarget)` aims the camera in that direction. ✓

- **Tilt-clamp gating via view-direction spherical is correct.** `MIN_SPHERICAL_PHI = (90 − MAX_TILT_DEGREES) * DEG2RAD ≈ 1°`; `MAX_SPHERICAL_PHI ≈ 179°`. For `offsetView = −fwd`: `offsetView.y = −fwd.y = sin(tiltDown)`, so `phi = acos(sin(tiltDown))`. At `tiltDown = +89°`: `phi ≈ 1°` = MIN. At `tiltDown = −89°`: `phi ≈ 179°` = MAX. Mapping is correct and matches the existing convention in `ExperimentalControls.js:58-61`. ✓

- **`this.center` API contract is preserved (no consumer breakage).** Confirmed: `_zoomActionBar` (line 1119) uses `camera.position.distanceTo(this.center)` — distance-to-rotation-centre is the right scaling for ActionBar zoom regardless of whether the camera is looking at centre. Plan handles this correctly.

- **Spec-drift items are minor.** The proposed §"Architecture additions" sentence in `001-phase-2-plan.md` and the new `decisions.md` entry both describe the same fix in different registers (architectural note vs. design-decision log entry). No duplication concern — they serve different readers. The `decisions.md` entry should also link forward from the 2026-05-10 entry ("latched rotation centre") so a future reader sees both decisions form a pair: *what* the centre is (latched), and *how* the orbit math uses it (decoupled view).

## Bottom line

The plan is mathematically sound and the implementation is small. The one design call worth pushing back on (gently) is whether dual-spherical is the right *behavioural* fix, or whether the user would prefer a more conservative "delay the first `lookAt(center)` by one frame" tweak that preserves orbit-feel. Worth a quick check with the user before the ~10-line code change lands. Everything else is either confirmed correct or a one-sentence spec polish.
