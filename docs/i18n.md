# Internationalization (i18n)

Tracking issue: [#656](https://github.com/3DStreet/3dstreet/issues/656).

3DStreet's editor UI can render in multiple languages via
[`react-intl`](https://formatjs.io/docs/react-intl/). Today we ship **English
(`en`)**, **Spanish (`es`)** and **Brazilian Portuguese (`pt-BR`)**. The feature
is a cheap, mostly-automated acquisition/activation experiment shipped **behind
a flag** and measured in PostHog — not a big-bang localization project.

## How it works at runtime

- `src/editor/i18n/config.js` — the source of truth for supported locales,
  browser-language detection, the feature flag, and the initial-locale
  resolution. While the flag is off the app is forced to English, so wrapping
  strings is always safe.
- `src/editor/i18n/LocaleProvider.jsx` — an `IntlProvider` driven by the
  `locale` value in the Zustand store (`src/store.js`). Mounted at the React
  roots in `src/editor/index.jsx`.
- `src/editor/components/elements/LanguagePreference` — the manual language
  switcher in the ActionBar. Hidden unless the flag is enabled.

### Feature flag

The experiment is gated by the PostHog feature flag `localization`. For local
development / QA (PostHog is disabled in dev) use either override:

- URL: `?i18n=true` (or `?i18n=false`)
- localStorage: `localStorage.setItem('i18nEnabled', 'true')`

Resolution order: URL → localStorage → PostHog flag → off.

### PostHog instrumentation

Every event is tagged with a `locale` super property (registered at startup in
`index.jsx` and on each switch in `store.js`), and a `locale_changed` event
fires when the user picks a language. This is what lets us compare
activation/retention/conversion across `en` / `es` / `pt-BR` cohorts.

## Adding / changing strings

Wrap user-facing strings with `react-intl`, always providing an explicit `id`
and a `defaultMessage` (the English text — the canonical source):

```jsx
import { FormattedMessage, useIntl } from 'react-intl';

// JSX children:
<FormattedMessage id="newModal.title" defaultMessage="Create a new scene" />

// String props / JS expressions:
const intl = useIntl();
el.title = intl.formatMessage({ id: 'actionBar.zoomIn', defaultMessage: 'Zoom In' });

// Interpolation:
<FormattedMessage
  id="scenesModal.count"
  defaultMessage="You have {count} scenes"
  values={{ count }}
/>
```

IDs are readable and namespaced by area, e.g. `newModal.*`, `actionBar.*`,
`geoSidebar.*`.

> Note: the formatjs extractor parses `.jsx` (and plain `.js`) files. A handful
> of legacy property-editor widgets put JSX in `.js` files
> (`src/editor/components/widgets/*.js`, some `*Sidebar.js`); the extractor skips
> those, so they are out of scope until renamed to `.jsx`.

## Catalogs & the translation pipeline

Catalogs live in `src/editor/i18n/locales/`:

- `en.json` — **generated** by extraction; the English source of truth.
- `es.json`, `pt-BR.json` — **generated** by translation.
- `.translation-manifest.json` — bookkeeping: the English source each key was
  last translated from, used to detect changed strings.

Commands:

```bash
npm run i18n:extract     # scan source → regenerate en.json (formatjs)
npm run i18n:translate   # translate missing/changed keys → es.json, pt-BR.json
npm run i18n:check       # report drift, non-zero exit if catalogs are stale (CI)
```

`npm run i18n:translate` calls the Anthropic Messages API and needs
`ANTHROPIC_API_KEY` (model overridable via `I18N_MODEL`, default
`claude-sonnet-4-6`). It only re-translates keys that are new or whose English
source changed, prunes keys removed from `en.json`, and keeps urban-planning
terminology consistent via a glossary in the script.

### CI

`.github/workflows/i18n.yml`:

- **Pull requests** run `i18n:check` and fail on drift (no API key needed).
- **Pushes to `main`** run `i18n:translate` (using the `ANTHROPIC_API_KEY`
  secret) and commit refreshed catalogs, so locales never drift.

## Typical workflow

1. Wrap new strings with `FormattedMessage` / `useIntl` + a stable `id`.
2. `npm run i18n:extract` to update `en.json`.
3. `npm run i18n:translate` to fill in `es.json` / `pt-BR.json` (or let the
   main-branch CI job do it).
4. Commit all of `src/editor/i18n/locales/`.
